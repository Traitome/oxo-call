## Concepts

- **Input requirement**: Bracken reads the output of `kraken` (default `--report` flag) or `kraken2`, which is a flat classification file where each read is assigned to a single taxonomic node. It does NOT accept raw FASTQ/FASTA files; Kraken must be run first to generate the classification report.
- **Database dependency**: Bracken requires the same Kraken database that was used for classification. The database must include Bracken-generated hash files (created by `bracken-build` using the `--kmer` and `--readlen` values that match the sequencing data).
- **Abundance reestimation**: Rather than simply counting classified reads, Bracken uses a Bayesian model to redistribute unclassified or ambiguously-assigned reads back to taxa based on the probability of observing a given k-mer from each species at the measured read length.
- **Threshold control**: The `-t`/`--threshold` parameter sets the minimum number of reads a species must have (at the specified rank) before its abundance is reported. Raising this threshold filters low-abundance taxa and reduces false positives.
- **Rank flexibility**: The `-l`/`--level` parameter specifies the taxonomic rank for reporting: `S` (species, default), `G` (genus), `F` (family), `O` (order), `C` (class), `P` (phylum), `K` (kingdom). Abundance estimates are reestimated at the chosen rank, not propagated upward.

## Pitfalls

- **Kmer/read-length mismatch**: Specifying a `--readlen` value that does not match the actual average read length of the sequenced data produces incorrect abundance reestimation. Using `--readlen` that is too short inflates estimates for species with shorter classified reads; too long inflates estimates for long-read data.
- **Database version mismatch**: Running `bracken-build` with a different database or k-mer size than the one used for Kraken classification breaks the model. The hash file generated by `bracken-build` must correspond exactly to the Kraken database used for classification.
- **Confusing threshold with count**: The `-t`/`--threshold` value filters the *minimum read count per taxon at the chosen rank*, not a percentage or proportion. Setting `-t 10` means a genus must have at least 10 genus-level reads to appear in output, which is more stringent for high-depth datasets than it may appear.
- **Ignoring the abundance matrix for large studies**: When analyzing many samples, repeatedly reestimating abundances from raw Kraken reports is inefficient. The intermediate `.kreport` file can be reused across multiple `bracken` estimation runs without re-running Kraken.
- **Omitting `-d` for non-standard database paths**: If the Kraken database is not installed in the directory referenced during the original `kraken` run, the `-d`/`--database` flag must be provided to `bracken` or `bracken-build`, otherwise the tool looks for the default installation path and fails.

## Examples

### Estimate species-level abundances from a Kraken report
**Args:** `-d /path/to/kraken_db -i sample.kraken -o sample_abundance.tsv -r sample.kreport -l S`
**Explanation:** This reads the Kraken classification report for the sample and reestimates abundances at the species level, outputting a two-column TSV with species names and estimated read counts.

### Estimate genus-level abundances with a higher read threshold
**Args:** `-d /path/to/kraken_db -i sample.kraken -o genus_abundance.tsv -r sample.kreport -l G -t 20`
**Explanation:** Genus-level estimation aggregates species into their parent genera, and the threshold of 20 requires at least 20 genus-level classified reads for a genus to appear in output, reducing noise from low-coverage taxa.

### Build a Bracken k-mer probability matrix for a 150 bp read dataset
**Args:** `bracken-build -d /path/to/kraken_db -k 35 -l 150`
**Explanation:** This regenerates the k-mer probability distribution matrix using a k-mer size of 35 (standard for Kraken2) and a read length of 150, which must match the library prep and sequencing length before running `bracken`.

### Generate combined species and genus reports in a single pipeline
**Args:** `-d /path/to/kraken_db -i sample.kraken -o sample_combined.tsv -r sample.kreport -l S -t 10 && bracken -d /path/to/kraken_db -i sample.kraken -o sample_genus.tsv -r sample.kreport -l G -t 5`
**Explanation:** Two sequential Bracken calls with different rank settings generate independent output files; species-level estimation and genus-level estimation are independent and cannot be merged in a single call.

### Reestimate abundances for multiple samples using a shared Kraken database
**Args:** `-d /path/to/kraken_db -i batch1.kraken -o batch1_abundance.tsv -r batch1.kreport -l S -t 10`
**Explanation:** Because the Kraken database is shared, `bracken-build` only needs to be run once per database; subsequent samples reuse the same probability matrix, making this efficient for processing multiple libraries in parallel.

### Pipe Kraken2 output directly to Bracken for species abundance in real-time analysis
**Args:** `bracken -d /path/to/kraken_db -i /dev/stdin -o stdout.tsv -r stdin.kreport -l S -t 5`
**Explanation:** Bracken can read from stdin when Kraken2 outputs its classification report via a pipe, but the `.kreport` file must also be provided or the reestimation lacks the read length distribution needed for the Bayesian model.

### Export a summary TSV for downstream statistical analysis in R or Python
**Args:** `-d /path/to/kraken_db -i sample.kraken -o sample_species.tsv -r sample.kreport -l S -t 2`
**Explanation:** A low threshold of 2 captures rare species present at minimal abundance, which is useful when the goal is exhaustive species lists for downstream diversity analyses such as alpha-diversity or differential abundance testing.
