## Concepts

- Bustools operates on BUS file format (binary), which stores pseudoaligned reads with their equivalence class assignments, transcript IDs, and UMI sequences. The format is designed for efficient processing of single-cell transcriptomes.
- The tool requires a kbus index generated by `kallisto bus` and must process BUS files through specific stages: sorting (by default by UMI then by barcode) before counting, and correcting for errors if needed.
- Bustools supports multiple subcommands including `inspect` (view file statistics), `sort` (reorder records), `text` (convert to human-readable format), `merge` (combine multiple samples), `correct` (error correction), and `count` (generate count matrices).

## Pitfalls

- Forgetting to sort the BUS file before running `bustools count` will cause the counting to fail or produce incorrect results, as count operations require records sorted by UMI then barcode.
- Using an incompatible kbus index (generated with different parameters or kallisto version) with the BUS file will lead to erroneous equivalence class assignments and incorrect count matrices.
- Not specifying the correct EC (equivalence class) format when generating counts can produce ambiguous or incomplete gene-level count matrices, especially when transcripts map to multiple genes.
- Attempting to process extremely large BUS files without sufficient RAM will cause memory allocation errors, as some operations load significant portions into memory.
- Running `bustools correct` without an appropriate whitelist can introduce artifacts rather than true error corrections, worsening data quality.

## Examples

### Inspecting a BUS file to verify its contents and structure
**Args:** `inspect input.bus`
**Explanation:** Displays metadata including number of records, barcode/UMI statistics, equivalence class distribution, and flags indicating whether the file is sorted.

### Sorting a BUS file by UMI and barcode for downstream counting
**Args:** `sort -o sorted.bus input.bus`
**Explanation:** Rearranges the binary BUS records in sorted order required by the count command; this is a prerequisite step before generating count matrices.

### Generating a count matrix (gene-level) from a sorted BUS file
**Args:** `count -g genes.txt -o counts_matrix sorted.bus`
**Explanation:** Aggregates UMIs to genes using the provided gene annotation file, producing a matrix where rows are genes and columns are cells. Requires a sorted BUS file.

### Converting a BUS file to human-readable text format
**Args:** `text input.bus output.txt`
**Explanation:** Exports the binary BUS records as tab-separated text with columns for barcode, UMI, EC, and count, useful for debugging or custom analysis.

### Merging multiple BUS files from different cells or batches
**Args:** `merge -o merged.bus sample1.bus sample2.bus sample3.bus`
**Explanation:** Combines multiple BUS files into a single file, useful for aggregating data across multiple runs before downstream analysis.

### Generating transcript-level count matrix instead of gene-level
**Args:** `count -g genes.txt -o trans_counts -t sorted.bus`
**Explanation:** Produces a count matrix at the transcript level rather than gene level, preserving isoform information for analysis of alternative splicing.

### Filtering BUS records by gene/isoform using an allowed list
**Args:** `filter -o filtered.bus input.bus -c genes.txt`
**Explanation:** Retains only records where the equivalence class overlaps with genes in the provided allowlist, removing unwanted transcripts or contamination.
