## Concepts

- **Input data model**: CONSEL requires two input files — a tree file containing candidate topologies in Newick format (one tree per line) and a likelihood data file with sitewise log-likelihood values for each tree. The likelihood file is typically generated by phylogenetic packages like PAML (codeml) or other ML tree estimators.
- **Statistical tests available**: CONSEL implements four hypothesis tests — Kishino-Hasegawa (KH), Shimodaira-Hasegawa (SH), weighted Shimodaira-Hasegawa (WSH), and Approximately Unbiased (AU) test. Each test provides p-values for assessing whether a given tree is significantly worse than the best-scoring topology.
- **Bootstrap resampling**: The AU and WSH tests use multisample bootstrap or parametric bootstrap resampling to obtain calibrated p-values. The number of bootstrap replicates (`-boot` option) should be ≥10000 for reliable p-values, especially for the AU test.
- **Output interpretation**: Output is a table with columns for test name, likelihood difference (delta-L), and p-value. Trees with p-value ≥0.05 (or ≥0.95 for AU) are considered not significantly different from the best tree at the 95% confidence level.

## Pitfalls

- **Using insufficient bootstrap replicates**: Running CONSEL with fewer than 10000 bootstrap replicates can produce unreliable p-values, especially for the AU test which is sensitive to sample size. This leads to incorrect confidence assessment of tree topologies.
- **Misinterpreting AU test confidence sets**: The AU test returns the confidence set as trees with p-value ≥0.05 (not ≤0.95). Treating it as "best 95%" will exclude valid trees. Always use the 0.05 threshold forAU p-values.
- **Ignoring multiple testing**: When comparing many alternative trees simultaneously, the family-wise error rate increases. CONSEL tests each tree independently without correction, so declaring all trees with p > 0.05 as equally valid may yield false positives.
- **Using trees with identical likelihoods**: If two trees have identical sitewise likelihood profiles, CONSEL will produce undefined or zero-variance results. Ensure that candidate trees represent genuinely different topologies.

## Examples

### Assess confidence of an alternative tree topology
**Args:** `-tree cand.tree -datLike cand.lik -boot 10000`
**Explanation:** Loads candidate trees from `cand.tree` and sitewise likelihoods from `cand.lik`, then performs 10000 bootstrap replicates to compute AU and WSH p-values for statistical confidence assessment.

### Run only the KH and SH tests without bootstrapping
**Args:** `-tree cand.tree -datLike cand.lik -nobootsh -nokhch`
**Explanation:** Computes KH and SH test statistics using the asymptotic chi-squared distribution without bootstrap resampling, suitable for quick preliminary analysis.

### Specify output file for results
**Args:** `-tree cand.tree -datLike cand.lik -out result.txt`
**Explanation:** Writes all test results to `result.txt` instead of printing to stdout, allowing later inspection or integration into pipelines.

### Use parametric bootstrap for tree comparison
**Args:** `-tree cand.tree -datLike cand.lik -boot 50000 -pboot`
**Explanation:** Performs 50000 parametric bootstrap replicates for more accurate AU test p-values when the phylogenetic model assumptions may not hold for sitewise resampling.

### Compare trees without sitewise likelihood file (using precomputed statistics)
**Args:** `-tree cand.tree -ml -out stats.txt`
**Explanation:** Runs a simplified analysis using only maximum likelihood scores if full sitewise likelihoods are unavailable; outputs results to `stats.txt` for review.
