## Concepts

- **Trio phasing dependency**: clair3-trio requires pre-phased BAM files generated by WhatsHap (or similar haplotypers). The phasing information is embedded in the `HP` tag of the BAM records. Without properly phased reads, trio co-analysis cannot function, and the pipeline will exit or produce unreliable output.

- **Sample order convention**: The `--ctA`, `--ctB`, and `--ctC` arguments must specify the child, father, and mother BAMs respectively. Mixing up this order leads to incorrect inheritance annotation and potentially harms downstream genetic counseling interpretation.

- **Joint haploid representation model**: Unlike singletons that use diploid pileup, trio mode activates haploid pileup representation for each allele, enabling the model to exploit inheritance constraints (Mendelian transmission) and dephase reads for improved sensitivity near challenging regions like short tandem repeats.

- **Output VCF inheritance tagging**: The emitted VCF file includes INFO tags such as `INHERITANCE` (auto-recoded from trio reference model) and per-sample genotype annotations (`GT`, `AD`, `GQ`) that reflect trio-aware likelihoods, not independent diploid estimates.

- **Resource scaling with contigs**: For GRCh38 or genomes with many contigs, memory consumption scales with `--var_min_alt_frac_window`. Running on a subset of chromosomes via `--include_alleles_file` reduces peak RAM but must be coordinated across all trio samples.

## Pitfalls

- **Unphased BAM input causes silent failures**: Providing non-phased BAM files (e.g., from aligners that do not set HP tags) produces no error message in early stages but leads to degenerate phasing results and poor variant sensitivity. Always verify phasing with `whatshap stats` before running.

- **Reference mismatch between BAM and FASTA**: If the BAM header `@SQ` lines reference a different genome version or contig naming convention than the `--ref` FASTA, the read-to-reference alignment step produces mapping anomalies and the model sees corrupted pileup entries.

- **Insufficient GPU memory for large genomes**: On GRCh38 with default pileup parameters, VRAM requirements exceed 10 GB. Running on a GPU with less memory causes OOM errors. Reduce `--var_min_alt_frac_window` or use `--chunk_size` to limit concurrent processing.

- **Incorrect chromosome naming convention**: Some pipelines strip the `chr` prefix from contig names in BAM files. Using a reference genome with `chr` prefixes while BAM files lack them (or vice versa) produces zero reads in pileup and empty VCF output.

- **Ignoring return code for partial failures**: The pipeline may exit with code 0 even when certain contigs are skipped due to read coverage thresholds. Always check the printed summary lines and confirm expected variant counts per chromosome.

## Examples

### Call variants for a family trio with default Nanopore settings
**Args:** `--ref grch38.fa --trioFathers father.phased.bam --trioMothers mother.phased.bam --ctA child.phased.bam --model_path models/ont-trio --threads 16 --output output_dir`
**Explanation:** This specifies all three trio BAMs (pre-phased by WhatsHap), the reference genome, a trio-aware ONT model, and output directory, enabling joint inference with default pileup parameters and parallelization.

### Call variants using a subset of chromosomes for speed
**Args:** `--ref grch38.fa --ctA child.bam --ctB father.bam --ctC mother.bam --model_path models/ont-trio --output trio_out --include_alleles_file chr_list.txt --threads 32`
**Explanation:** Restricting processing to chromosomes listed in `chr_list.txt` (one per line) dramatically reduces runtime and memory, useful for targeted validation or when trio data exists for specific chromosomes only.

### Call with reduced GPU memory footprint
**Args:** `--ref hg38.fa --ctA child.bam --ctB father.bam --ctC mother.bam --model_path models/ont-trio --output trio_out --var_min_alt_frac_window 1 --chunk_size 50 --threads 8`
**Explanation:** Lowering `--var_min_alt_frac_window` reduces per-allele context width, cutting VRAM usage while still producing valid calls. `--chunk_size 50` limits concurrent contigs processed simultaneously, also easing memory pressure.

### Call using a PacBio HiFi model with BAM input validation
**Args:** `--ref grch38.fa --ctA child.hifi.bam --ctB father.hifi.bam --ctC mother.hifi.bam --model_path models/hifi-trio --output trio_hifi --threads 24 --use_supplementary_for_bam_regenerate`
**Explanation:** Activating the HiFi trio model with `--use_supplementary_for_bam_regenerate` extends alignment regeneration to include supplementary alignments, which is standard for HiFi reads but must be explicitly enabled for complete pileup generation.

### Call with external phased VCF for read-back phasing
**Args:** `--ref grch38.fa --ctA child.bam --ctB father.bam --ctC mother.bam --model_path models/ont-trio --output trio_out --phased_vcf reference_phasing.vcf --threads 16`
**Explanation:** Providing an external phased VCF via `--phased_vcf` overrides WhatsHap-derived phasing for read disambiguation, which is useful when a high-confidence reference panel is available and WhatsHap phasing is sparse or erroneous.
