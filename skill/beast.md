## Concepts

- BEAST operates on XML input files generated by BEAUti (BEAST User Interface), which define the sequence alignment, substitution model, clock model, tree prior, and MCMC chain parameters. Running `beast` directly with a properly configured XML file executes the full Bayesian analysis pipeline.
- Output is written to two primary file types: `.log` files containing parameter samples drawn from the posterior distribution (one sample per sampling interval), and `.trees` files containing phylogenetic trees annotated with estimated node heights and posterior probabilities.
- The framework supports multiple substitution models (HKY, TN93, GTR), clock models (strict, relaxed log-normal, random local), and tree priors (constant size, exponential growth, Bayesian skyline, birth-death), which must be biologically appropriate for the data.
- Posterior probability distributions are approximated through MCMC sampling; convergence requires sufficiently long chain lengths and adequate burn-in periods to ensure the chain has mixed properly before drawing from the stationary distribution.
- Parameter effective sample size (ESS) values in output logs indicate sampling quality; ESS values below 200 typically suggest insufficient mixing or chain length, signaling the need for longer runs or adjusted operators.

## Pitfalls

- Setting the chain length too short produces unreliable posterior estimates with high statistical uncertainty, as the MCMC chain fails to explore the parameter space adequately before sampling ends.
- Using an inappropriate substitution model (for example, applying JC69 to data with significant nucleotide composition bias) systematically biases phylogenetic estimates and branch length calibrations.
- Discarding insufficient burn-in samples allows early parameter states from the non-stationary phase to contaminate posterior summaries, leading to biased mean and credibility interval estimates.
- Specifying unrealistic calibration constraints (such as conflicting node age priors or absurdly narrow credible intervals) causes the MCMC to struggle, resulting in poor acceptance rates, stuck chains, or mode-switching behavior that invalidates results.
- Failing to account for taxonomic sampling effects by ignoring tip dating information when present produces incorrectly scaled molecular clock rates and node age estimates.

## Examples

### Estimate phylogeny with HKY substitution and strict clock
**Args:** `input.xml -threads 4`
**Explanation:** Runs a Bayesian phylogeny estimation using the HKY substitution model and strict molecular clock on 4 CPU threads, sampling from the posterior until the chain completes.

### Run molecular clock dating with calibration priors
**Args:** `input.xml -overwrite`
**Explanation:** Executes a dating analysis with tip date calibrations defined in the XML file, overwriting any existing output files with fresh results.

### Perform relaxed clock analysis with log-normal rates
**Args:** `input.xml -seed 42 -beagle_gpu`
**Explanation:** Runs a relaxed clock analysis that allows rate variation among lineages, using a reproducible random seed and GPU acceleration via BEAGLE library for faster likelihood computation.

### Generate skyline plot for population dynamics
**Args:** `input.xml -statefile state.txt -log 5`
**Explanation:** Performs a demographic reconstruction using the Bayesian skyline plot model, saving the state to a checkpoint file every 5 million states for resumable long-running analyses.

### Resume an interrupted BEAST analysis
**Args:** `input.xml -statefile state.txt -resume`
**Explanation:** Resumes a previously checkpointed MCMC run from the saved state file, continuing sampling where the previous run stopped rather than starting over.
