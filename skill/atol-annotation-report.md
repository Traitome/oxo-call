## Concepts

- **Input format**: `atol-annotation-report` reads ATOL annotation files (TSV/CSV) produced by the ATOL annotation pipeline, and optionally accepts raw VCF files for cross-reference annotation enrichment. The input file must follow the ATOL column schema (CHROM, POS, ID, REF, ALT, ann.gene, ann.consequence, ann.csq, ann.max_af, ann.am_path, ann.clin_sig, and ATOL-specific scoring columns) or an error is raised.
- **Output formats**: The tool supports multiple output formats controlled by `--format` (or `-F`): `html` (default, interactive report with filtering UI), `csv` (flat table for spreadsheet import), `json` (structured data for downstream pipelines), and `vcf` (annotated VCF with INFO field additions). The choice of output format affects which columns are included in the output.
- **Filtering model**: Rows are filtered before report generation using `--filter-gene`, `--filter-consequence`, `--filter-csq`, `--max-af`, `--min-score`, `--keep-only-pathogenic`, and `--exclude-clin-sig`. Filters are applied in a logical AND fashion. Using `--filter-consequence` with a consequence string (e.g., `missense_variant`) requires exact matching against the ann.consequence field, so partial strings cause silent mismatches.
- **Column selection**: `--columns` (or `-c`) overrides the default column set. Columns are specified by their ATOL schema names, not display names. Using a column name that does not exist in the input schema silently produces an empty column in the output rather than a warning.
- **Reporting thresholds**: The `--significance-threshold` and `--score-cutoff` flags control what is flagged in the report header and summary statistics. These values do not automatically propagate to row-level filters; they are purely for flagging and labeling purposes.

## Pitfalls

- **Mismatched input schema**: Providing an input file generated by a different annotation tool or an older ATOL version that uses a different column naming convention (e.g., `ann.gene` vs `Gene` vs `GENE`) causes columns to be read as empty, producing a report full of blank cells. Always verify the input file's header row matches the expected ATOL schema before running.
- **Filter logic is AND, not OR**: Specifying multiple `--filter-consequence` values with the expectation that they will be ORed together results in zero rows being returned. To achieve OR logic across consequences, use `--filter-csq` with a properly formatted CSQ field string and confirm it is supported in your ATOL version.
- **Overwriting outputs without `--force`**: Running the tool with an output path that already exists causes the tool to abort with an error rather than appending or prompting. This is especially problematic in batch workflows where the same output directory is reused; each invocation must include `--force` or a unique output path.
- **Large input files without `--batch-size`**: For inputs exceeding ~50,000 variants, the tool loads the entire file into memory by default, which can cause an out-of-memory crash on systems with limited RAM. Setting `--batch-size` to a smaller integer (e.g., `10000`) processes the input in chunks, reducing memory usage.
- **HTML output file size with many columns**: Including all columns via `--columns all` in an HTML report on a large input can produce extremely large files (hundreds of MBs), making the browser-based report slow to open. Use `--columns` to select only required columns and prune unnecessary fields.

## Examples

### Generate an HTML annotation report from an ATOL annotation file
**Args:** `--input variants.annot.tsv --output report.html`
**Explanation:** This produces an interactive HTML report using all default columns and the default filtering, which is appropriate for a first-pass review of a newly annotated variant set.

### Export a CSV report filtered to only high-confidence pathogenic variants
**Args:** `--input variants.annot.tsv --format csv --output path_variants.csv --keep-only-pathogenic`
**Explanation:** The `--keep-only-pathogenic` flag restricts output to rows where the ann.am_path column is flagged, and `--format csv` produces a flat table suitable for import into Excel or R for downstream statistical analysis.

### Generate a JSON report for programmatic downstream processing
**Args:** `--input variants.annot.tsv --format json --output report.json --min-score 0.8 --force`
**Explanation:** The `--min-score 0.8` threshold removes low-confidence annotations before output, and `--force` allows the output JSON to overwrite any pre-existing file at the target path without prompting.

### Generate a report restricted to missense and splice-site variants in a specific gene
**Args:** `--input variants.annot.tsv --output brca1_report.html --filter-gene BRCA1 --filter-consequence missense_variant --filter-consequence splice_site_variant`
**Explanation:** Combining `--filter-gene` with two `--filter-consequence` flags limits the report to only missense and splice-site variants affecting BRCA1, which is useful for focused clinical review of a gene of interest.

### Create a VCF report with all columns, suitable for integration back into a variant calling pipeline
**Args:** `--input variants.annot.tsv --format vcf --output annotated_output.vcf --columns all`
**Explanation:** Specifying `--columns all` ensures every annotated field from the ATOL schema is embedded into the VCF INFO field, enabling the annotated VCF to be fed back into downstream tools like bcftools or vcfanno.

### Generate a batch-processed CSV report from a large file using chunked processing
**Args:** `--input large_cohort.annot.tsv --format csv --output cohort_report.csv --batch-size 20000 --force`
**Explanation:** The `--batch-size 20000` flag splits the input into 20,000-row chunks processed sequentially, keeping peak memory usage manageable for very large cohort annotation files, while `--force` ensures the output is overwritten if it already exists.
